SKINNY-64-128 OPTIMIZATIONS IMPLEMENTED
=======================================

This file verifies requested SKINNY optimizations and documents all
implemented optimizations in current CPU/GPU code.

1) REQUESTED CHECKLIST STATUS
=============================

1.1 S-box (4-bit)
-----------------

- Constant-memory lookup S-box on GPU: IMPLEMENTED
	- GPU table path uses 16-entry S-box from constant memory (`cuda.const.array_like`).

- Bitsliced logic S-box on GPU: IMPLEMENTED
	- GPU bitsliced path uses fixed boolean equations over packed nibble lanes.

- GF(2^4) arithmetic S-box implementation: NOT IMPLEMENTED
	- Current compute path is bitsliced boolean logic, not explicit field arithmetic inversion.

- Recommendation (either table or bitslice): IMPLEMENTED
	- Both paths exist and are selectable via `variant`.


1.2 Linear Layer (ShiftRows + MixColumns equivalent)
-----------------------------------------------------

- Compute linear layer on-the-fly using bit operations: IMPLEMENTED
	- `round_linear_dev` uses row rotations and XOR mixing only.
	- No precomputed linear-layer tables are used.

- Precomputed table approach for linear layer: NOT IMPLEMENTED
	- The implementation intentionally uses direct compute.


1.3 AddRoundKey
---------------

- Per-round XOR with precomputed key material: IMPLEMENTED
	- Round subkeys are precomputed and XORed into state each round.

- Keep keys near compute path: IMPLEMENTED
	- GPU stages 36 round subkeys into shared memory at kernel start.


1.4 CPU Baseline (SKINNY)
-------------------------

- Numba/JIT CPU baseline with tight loops: IMPLEMENTED
	- CPU uses njit-compiled loops and packed uint64 state processing.

- CPU table-based S-box baseline: IMPLEMENTED
	- CPU table variant is implemented and used in benchmark baseline.

- CPU bitsliced variant: IMPLEMENTED
	- CPU bitsliced S-box path is available as selectable variant.

- Explicit round unrolling / SIMD intrinsics baseline: NOT IMPLEMENTED
	- Current Python/Numba path uses loops; no manual unroll or C intrinsics.


2) ALL IMPLEMENTED OPTIMIZATIONS
================================

2.1 S-box Optimizations
-----------------------

- Dual S-box strategies on CPU and GPU
	- Table variant for simple low-footprint lookup.
	- Bitsliced variant for table-free compute pipeline.
	- Benefit: supports architecture-specific performance tuning and comparisons.

- Constant-memory 16-entry S-box table (GPU table variant)
	- Benefit: tiny lookup footprint with cached constant-memory access.

- Bitsliced boolean-network S-box (CPU and GPU)
	- Benefit: removes table fetches and keeps nonlinear layer in ALU/register path.


2.2 Linear Layer Optimizations
------------------------------

- Register-level ShiftRows and MixColumns-equivalent transform
	- Implemented via 16-bit row rotations and XOR combinations.
	- Benefit: avoids memory lookups and keeps linear layer deterministic and fast.

- Compact row-oriented state decomposition
	- 64-bit state split into 4 x 16-bit rows, transformed, then re-packed.
	- Benefit: efficient implementation of SKINNY linear operations.


2.3 AddRoundKey and Key-Schedule Optimizations
----------------------------------------------

- Precomputed round subkeys (36 rounds)
	- Subkeys generated once per key via `generate_round_subkeys`.
	- Benefit: removes key schedule overhead from round hot loops.

- CPU round-subkey cache by active key
	- Benefit: avoids recomputing subkeys for repeated same-key calls.

- GPU shared-memory subkey staging
	- Each block loads subkeys once into shared memory.
	- Benefit: reduces repeated global-memory subkey accesses.


2.4 Execution-Level Optimizations
---------------------------------

- GPU thread coarsening
	- Each thread processes 2 blocks.
	- Benefit: improved latency hiding and lower per-block launch overhead.

- CPU parallel block processing
	- Numba `parallel=True` with `prange` across independent blocks.
	- Benefit: multi-core scaling for ECB/CTR workloads.

- CPU worker clamping and restoration
	- Worker count is bounded by logical cores; previous thread setting restored afterward.
	- Benefit: avoids oversubscription and prevents persistent global thread-state side effects.


2.5 Data Layout and State Handling Optimizations
------------------------------------------------

- Packed uint64 block representation
	- Blocks processed as uint64 arrays on CPU and GPU.
	- Benefit: compact state operations and efficient bitwise transforms.

- Low-word selective AddRoundKey update
	- Round subkey XOR is applied on low 32 bits, then merged back with upper state.
	- Benefit: mirrors SKINNY round structure with minimal extra operations.


2.6 Mode and Utility Optimizations
----------------------------------

- First-class CTR support in CPU and GPU classes
	- CTR implemented as ECB(counter blocks) plus XOR.
	- Benefit: consistent mode API and shared semantics across backends.

- Shared CTR helper utilities
	- Counter generation and XOR centralized in shared utility module.
	- Benefit: reduced duplication and consistent correctness behavior.


2.7 Correctness and Benchmarking Hardening
------------------------------------------

- Input validation in public encryption APIs
	- Enforces 8-byte block alignment and 16-byte key length.
	- Benefit: fail-fast behavior and safer integration.

- CPU/GPU output equivalence check in benchmark workflow
	- Benchmark raises on mismatch before recording results.
	- Benefit: prevents silent correctness regressions.

- Warm-up before timed runs
	- CPU and GPU paths are executed once before collecting timings.
	- Benefit: reduces cold-start/JIT noise.

- Median-based timing aggregation
	- Uses median over repeated runs for CPU and GPU timing series.
	- Benefit: more stable metrics under runtime variance.
