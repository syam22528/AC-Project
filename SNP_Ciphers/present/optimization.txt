PRESENT-128 OPTIMIZATIONS IMPLEMENTED
========================================

This file verifies the requested optimization checklist and documents all
implemented optimizations in the current PRESENT CPU/GPU code.

1) REQUESTED CHECKLIST STATUS
=============================

1.1 S-box (4-bit SubCells)
--------------------------

- Constant-memory table S-box on GPU: IMPLEMENTED
	- GPU table path uses a 16-entry constant-memory S-box lookup.

- Bitsliced logic S-box on GPU: IMPLEMENTED
	- GPU bitsliced path uses boolean equations (AND/XOR/NOT-style bitwise logic), table-free.

- Bitslicing as recommended high-throughput path: IMPLEMENTED AS DEFAULT
	- Default GPU variant is bitsliced.

- Shared memory for 16-byte S-box table: NOT USED (by design)
	- Table path uses constant memory, which is appropriate for this very small LUT.


1.2 Permutation Layer (pLayer)
------------------------------

- Register-level bitwise pLayer in kernel: IMPLEMENTED
	- pLayer is computed with shifts, masks, and OR in device code.
	- No external permutation memory is used for per-state permutation.

- Bitslice-friendly pLayer behavior: IMPLEMENTED
	- In bitsliced GPU path, pLayer remains register-level bit operations.


1.3 AddRoundKey
---------------

- In-place XOR with round key per round: IMPLEMENTED
	- CPU and GPU both apply AddRoundKey as direct XOR on 64-bit state.

- Small round-key storage close to compute path: IMPLEMENTED
	- GPU kernels stage 32 round keys in shared memory once per block.


1.4 Key Schedule
----------------

- CPU precompute round keys then upload/use: IMPLEMENTED
	- Round keys are generated on CPU and uploaded to GPU via set_key().

- On-the-fly key schedule inside GPU kernel: NOT IMPLEMENTED
	- Current implementation does not generate round keys inside kernel.


1.5 CPU Baseline (PRESENT)
--------------------------

- Numba JIT tight-loop baseline: IMPLEMENTED
	- CPU encryption uses njit-compiled loops over uint64 states.

- Parallel CPU block processing: IMPLEMENTED
	- Uses parallel=True with prange and configurable worker count.

- Small 16-entry table S-box baseline: IMPLEMENTED
	- CPU nonlinear layer uses table lookup S-box.

- CPU bitsliced baseline variant: NOT IMPLEMENTED
	- CPU currently provides table-based S-box path only.

- Explicit round unrolling: NOT IMPLEMENTED
	- Rounds run in a for-loop (not manually unrolled).


2) ALL IMPLEMENTED OPTIMIZATIONS
================================

2.1 SubCells (S-box) Optimizations
----------------------------------

- Dual GPU S-box variants
	- Table variant: 16-entry constant-memory lookup.
	- Bitsliced variant: boolean-network tableless computation.
	- Benefit: supports ablation and architecture-aware tuning.

- Bitsliced as default GPU path
	- Default variant is bitsliced in PresentGpuOptimized.
	- Benefit: avoids S-box lookup memory latency at high parallelism.


2.2 pLayer Optimizations
------------------------

- In-register bit permutation on CPU and GPU
	- pLayer is implemented with fixed bit moves (shift/mask/or).
	- Benefit: low-overhead deterministic permutation with no table fetches.


2.3 AddRoundKey and Key-Schedule Optimizations
----------------------------------------------


- Precomputed round keys (32 x 64-bit)
	- Key schedule is generated once per key (PRESENT-128).
	- Benefit: no key schedule cost inside encryption rounds.


- CPU round-key cache by key
	- CPU class caches round keys for repeated encryptions with same key.
	- Benefit: avoids repeated schedule generation.

- GPU shared-memory round-key staging
	- Round keys are copied from device memory to shared memory once per block.
	- Benefit: reduces repeated global-memory key access across rounds.


2.4 Execution-Level Optimizations
---------------------------------

- Thread coarsening on GPU
	- Each CUDA thread processes 2 blocks.
	- Benefit: better latency hiding and lower launch overhead per block.

- CPU parallelization with worker control
	- Uses prange with runtime worker count.
	- Benefit: scales throughput on multi-core CPUs.

- Worker clamping and restoration
	- Worker count is clamped to available cores; previous thread setting is restored.
	- Benefit: avoids oversubscription and side effects on global thread state.


2.5 Data Representation Optimizations
-------------------------------------

- Packed uint64 block representation
	- Plaintext/ciphertext blocks are processed as big-endian uint64 arrays.
	- Benefit: efficient bitwise round operations on compact state representation.


2.6 Mode and Utility Optimizations
----------------------------------

- First-class CTR support in CPU and GPU classes
	- CTR implemented as ECB(counter blocks) plus XOR.
	- Benefit: shared semantics and reusable mode API.

- Shared CTR helper utilities
	- Counter construction and XOR logic centralized in common helpers.
	- Benefit: less duplication and consistent behavior across backends.


2.7 Correctness and Measurement Hardening
-----------------------------------------


- Input validation in public APIs
	- Enforces PRESENT-128 key size and block alignment.
	- Benefit: fail-fast integration safety.

- CPU/GPU correctness cross-check in benchmark flow
	- Benchmarks compare outputs and fail on mismatch.
	- Benefit: catches regressions during optimization work.

- Warm-up before timing
	- CPU and GPU are executed once before timed runs.
	- Benefit: reduces cold-start and JIT/setup skew.

- Median-based runtime measurement
	- Timing uses median over multiple runs.
	- Benefit: reduces outlier impact for stable comparisons.
