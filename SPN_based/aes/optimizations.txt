AES-128 OPTIMIZATIONS IMPLEMENTED (BY CIPHER COMPONENT)
=======================================================

This file lists implementation optimizations by AES core component.
Benchmark-process notes are intentionally excluded.

1) KEYEXPANSION OPTIMIZATIONS
=============================

- One-time expansion per key
  - AES-128 keys are expanded once into the 176-byte round-key schedule.
  - Benefit: avoids repeated key schedule work for every block.

- CPU expanded-key reuse
  - The CPU path caches expanded round keys for the active key.
  - Benefit: no redundant expansion when encrypting multiple buffers with the same key.

- GPU pre-expansion on host
  - Key expansion is performed host-side, then round keys are uploaded once per key set.
  - Benefit: removes key schedule computation from per-block GPU encryption.

- Constant-key specialization path (GPU)
  - For constkeys mode, kernels are specialized per key and cached by (variant, key bytes).
  - Benefit: avoids recompilation and enables constant-memory key access.


2) SUBBYTES OPTIMIZATIONS
=========================

- Shared-memory S-box path (GPU default)
  - The 256-byte S-box is staged into shared memory per thread block.
  - Benefit: lower lookup latency than repeated global-memory access.

- Tableless compute S-box path (GPU bitsliced variant)
  - SubBytes is computed with GF(2^8) inversion and affine transform, without lookup tables.
  - Benefit: removes table dependency and provides a compute-centric alternative.

- JIT-compiled substitution loops
  - SubBytes loops execute in compiled kernels (CPU JIT and CUDA device code).
  - Benefit: reduced interpreter overhead and predictable instruction flow.


3) SHIFTROWS OPTIMIZATIONS
==========================

- In-place row rotation
  - ShiftRows is implemented with direct byte swaps/rotations in local state arrays.
  - Benefit: avoids temporary matrix transforms and extra allocations.

- Fixed-index permutation
  - Row moves use deterministic index operations instead of dynamic logic.
  - Benefit: minimal control overhead and straightforward compiler optimization.


4) MIXCOLUMNS OPTIMIZATIONS
===========================

- Branchless xtime primitive
  - GF(2^8) multiply-by-2 uses branchless bitwise arithmetic.
  - Benefit: stable execution and lower branch divergence/penalty.

- XOR-reuse formulation per column
  - Intermediate column XOR values are reused across all 4 output bytes.
  - Benefit: fewer repeated operations than naive per-term recomputation.

- Compiled column loops
  - MixColumns runs inside JIT/CUDA compiled kernels over compact local state.
  - Benefit: improved throughput for round-heavy block processing.


5) ADDROUNDKEY OPTIMIZATIONS
============================

- Flat round-key layout with offset addressing
  - AddRoundKey uses contiguous round-key bytes and offset-based XOR loops.
  - Benefit: simple memory access pattern and low indexing overhead.

- Shared round-key staging (GPU global mode)
  - Round keys are copied to shared memory once per block before round processing.
  - Benefit: reduces repeated device-memory key fetches during rounds.

- Constant-memory round keys (GPU constkeys mode)
  - Key-specialized kernels read round keys from constant memory.
  - Benefit: efficient broadcast behavior for key bytes used by many threads.


6) CROSS-COMPONENT EXECUTION OPTIMIZATIONS
===========================================

- Thread coarsening in GPU kernels
  - Each CUDA thread processes 2 AES blocks.
  - Benefit: better latency hiding and lower per-block launch overhead.

- CPU block-level parallelism
  - Independent blocks are processed with parallel JIT loops.
  - Benefit: scales block throughput across available CPU cores.

- Final-round specialization
  - Round 10 omits MixColumns as required by AES.
  - Benefit: avoids unnecessary work in the last round.


7) ADDITIONAL NON-ROUND OPTIMIZATIONS
=====================================

- Hardware-backed CPU ECB path (when available)
  - CPU AES can use native crypto backend (Crypto.Cipher AES ECB).
  - Benefit: leverages optimized low-level implementation on supported systems.

- JIT-only software execution policy
  - Non-JIT software fallback is removed from the CPU software path.
  - Benefit: prevents accidental execution on slower interpreted paths.

- Keyed object reuse in CPU hardware path
  - Hardware cipher object is reused when the key is unchanged.
  - Benefit: reduces repeated backend setup overhead.

- First-class CTR mode in CPU and GPU classes
  - CTR is implemented as ECB(counter blocks) + XOR inside cipher APIs.
  - Benefit: mode support is integrated into the implementation, not external glue code.

- Shared CTR utility layer
  - Counter generation and XOR are centralized in shared helpers.
  - Benefit: less duplication and consistent behavior across CPU/GPU paths.

- Vectorized counter-block generation
  - CTR counter builder uses a NumPy fast path for common counter widths.
  - Benefit: lower Python-loop overhead for large CTR workloads.

- Explicit API input validation
  - Public methods enforce block alignment and key-size constraints.
  - Benefit: fail-fast behavior and reduced integration errors.

- Consistent CPU/GPU CTR semantics
  - CPU and GPU interfaces follow the same nonce/counter construction model.
  - Benefit: easier cross-backend verification and interchangeable usage.
